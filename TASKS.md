#### 2. Data Fetching
- [x] **2.1 — `select('*')` over-fetching across `lib/db.js` (Medium)** — Audited all 4 functions against every consumer:
  - **`getAllocations(slug)` / `getAllAllocations()`** — no change. The `allocations` table only has 6 columns and every consumer uses all of them; `select('*')` isn't over-fetching here.
  - **`getPortfolio(slug)`** — no change. The portfolio detail page alone displays nearly every `portfolio_stats` column (all rolling windows, crisis CAGRs, ulcer stats, etc.); union across its 5 consumers covers all but `current_value`. Not worth narrowing for one column.
  - **`getPortfolios()`** — narrowed from `select('*')` (~43 columns) to an explicit 30-column list, dropping `current_value, total_months, last_updated, description, m1_link, kofi_link, cagr_1yr, cagr_3yr, annualized_volatility, pct_profitable_months, best_month, worst_month, longest_drawdown_months`. **Important:** the kept list must match ScreenerClient's full `ALL_COLUMNS` picker, not just its *default-visible* columns — the first draft of this list only included default-on columns and would have broken the off-by-default toggles (Best Year, UPI, GFC/Dotcom CAGR, all rolling avg/high). Added a code comment in `lib/db.js` reminding future edits to add new screener columns/filters to this list too. Verified via `npm run build` + browser check on `/`, `/database`, and `/portfolio-screener` (including toggling on Best Year/UPI/GFC CAGR/1yr Avg/10yr High columns — all populated correctly).
- [ ] **2.2 — `getAllAllocations()` runs unconditionally on `/account` (Medium)** — Every force-dynamic `/account` load fetches the entire `allocations` table even for users with zero saved mixes. Fix: `mixes.length > 0 ? getAllAllocations() : Promise.resolve([])` inside the existing `Promise.all` — no waterfall reintroduced, skips the query for the common zero-mix case.
- [ ] **2.3 — Duplicate `getPortfolio(slug)` call per portfolio page (Medium)** — `generateMetadata()` and the page component in `app/portfolios/[slug]/page.js` both call `getPortfolio(slug)` independently — two Supabase round trips for the same row across all 76 SSG builds. Fix: wrap `getPortfolio` (and ideally `getAllocations`, `getMonthlyReturns`) in React's `cache()`.
- [ ] **2.4 — `getRelatedPortfolios()` full-table scan per portfolio build (Low)** — Round 2 fetches the entire `portfolio_strategies` table, repeated for each of the 76 portfolio pages at build time. Fix: wrap the strategies fetch in `cache()` so it's fetched once per build. Build-time only, low impact.
- [ ] **2.5 — `/builder` waterfall: auth → subscription/count → `getPortfolioNames()` (Medium)** — `getPortfolioNames()` has no dependency on the auth result but is awaited last, adding a full extra round trip on this force-dynamic page. Fix: kick off `getPortfolioNames()` alongside `supabase.auth.getUser()` in the initial `Promise.all`.
- [ ] **2.6 — `/builder` uses `getUser()` instead of `getSession()` (Low-Medium)** — `proxy.js` middleware already calls `auth.getUser()` and refreshes the session cookie for every request; `/account` was switched to `getSession()` in June 2026 to avoid the extra Auth-server round trip but `/builder` wasn't updated. Fix: mirror `/account` — `const { data: { session } } = await supabase.auth.getSession(); const user = session?.user;`
- [ ] **2.7 — `/compare` waterfall: `getPortfolioNames()` then per-slug fetches (Low-Medium)** — `await getPortfolioNames()` runs to completion before the `Promise.all` over `slugs.map(...)` begins, but the two are independent. Fix: run both concurrently via an outer `Promise.all`.
- [ ] **2.8 — Homepage / Database lack explicit revalidation — decision needed (Low)** — Neither `app/page.tsx` nor `app/database/page.js` export `revalidate`/`dynamic`; both are fully static (SSG), so Top Strategies rankings and `/database` listings only update on the next deploy. Mirrors the accepted `/leaderboard` tradeoff ("rebuilds with each deploy, aligns with monthly update cycle"). If that should apply site-wide, no change needed; if Nick expects `/` and `/database` to reflect Stage 2 promotions without a code push, add `export const revalidate = 86400` to both. **Needs a decision, not a default fix.**

- [ ] **Investigate and fix high Vercel ISR reads (June 2026, in progress)** — Root causes identified, fixes applied, monitoring in progress. (1) `/leaderboard` was the only explicit ISR page (`revalidate = 86400`). Removed — leaderboard is now fully static SSG and rebuilds with each monthly deploy, which matches the data update cadence exactly. (2) Cloudflare was bypassing its own cache for logged-in users because session cookies trigger Cloudflare's default cache-bypass behaviour — meaning every logged-in user request passed through to Vercel and counted as an ISR read. Fixed with a Cloudflare Cache Rule (Rules → Cache Rules → "Eligible for cache" via "Or" conditions), expanded to cover all public pages + `/api/correlation-matrix` (see CLAUDE.md Cloudflare section for full list). Free plan supports up to 5 cache rules; Cache Analytics not available on free plan.
  - **Refined understanding (per Vercel's official ISR docs, June 2026):** ISR has two cache layers. (1) Vercel's CDN cache — ephemeral, per-region, "minutes to hours" retention, evicted under memory pressure. Reads here are **free** and don't count toward the 1M/month limit. (2) The durable ISR cache — region-pinned, persists 31 days. Reads here cost "Read Units" (1 unit = 8KB) and ONLY happen on a CDN cache miss. So a "read" = Vercel's edge cache missed and fell back to durable storage, not "any page view."
  - **Why deploys spike reads:** per Vercel's docs, "each deployment uses its own ISR cache and does not reuse the cache from a previous deployment." A new deploy means Vercel's CDN is cold for every path — the first wave of traffic across all regions after a deploy all misses the CDN and reads from durable storage. This fully explains the 55K-reads-in-12h spike on a deploy day (≈440MB read from durable storage — consistent with a cold-cache wave across ~70 portfolio pages + tools across multiple regions).
  - **How the Cloudflare fix still helps:** every request Cloudflare now serves from its own edge cache never reaches Vercel at all, so it can't contribute to Vercel CDN misses or durable reads. Fewer requests reaching Vercel = fewer CDN-miss events = fewer durable reads, especially in the hours after a deploy while Vercel's CDN is still cold.
  - **Open item — Hobby plan overage behavior:** Vercel's ISR docs describe pricing/mechanics but not what happens specifically on the Hobby (free, no-payment-method) plan when the included 1M reads/month is exceeded. Still unresolved — would need Vercel's plan-limits page or support to confirm.
  - **Post-fix check (June 13):** ISR Reads chart still showed ~50-80K/day after the Jun 9 fixes — no clear improvement yet. Diagnosed via `curl -I` against live pages (e.g. `/portfolios/golden-butterfly-portfolio`): `cache-control: public, max-age=14400, must-revalidate` is present (Vercel is sending cacheable headers correctly), and Cloudflare returned `MISS`/`MISS`/`REVALIDATED` on three quick repeat requests — the cache rule is functioning, but hits aren't sticking consistently.
  - **Root cause — per-colo cache fragmentation:** Cloudflare caches independently per edge colo (no shared cache by default). With ~75 cacheable pages × dozens of global colos × ~6 refreshes/day (4hr TTL), every colo's first visitor for each page triggers an origin read — this alone can produce tens of thousands of reads/day even with the cache rule working perfectly.
  - **Fix applied (June 13):** enabled **Smart Tiered Cache Topology** in Cloudflare (Caching → Tiered Cache) — consolidates edge colos behind a tiered cache so only the tier needs to fetch from Vercel, not every colo independently. Free on all plans.
